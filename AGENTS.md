# Agent 工作区规范

本文件约束所有在本仓库中修改代码、运行测试或生成产物的 AI coding agent。

核心目标：**保持工作区干净、可追溯、可复现、易回滚，并满足 AAAI 论文实验的复现与审稿核验要求。**

---

## 1. 核心原则

1. 只修改完成当前任务所需的文件，不做无关重构、移动或大面积格式化。
2. 不得覆盖、回滚或删除用户已有改动，除非用户明确要求。
3. 新增文件必须用途明确、命名稳定，并放入符合项目现有结构的目录。
4. 不得将一次性脚本、调试输出、缓存或实验文件混入根目录和正式源码目录。
5. 不得为通过测试而绕过真实逻辑、硬编码结果、删除测试、跳过失败样本或降低断言强度。
6. 不得静默修改配置、依赖、lockfile、测试策略、数据定义或论文方法；必要修改须说明原因和影响。
7. 默认不修改生产数据、外部服务或已发布上游产物；涉及高影响操作时先 dry-run 或本地验证。
8. 任务结束前清理临时文件和调试代码，并报告改动与验证结果。

### 1.1 参考资产访问边界

- `legacy/` 和 `data/` 是参考资产，默认不得递归搜索、全量读取或作为常规上下文扫描对象。
- 仅当用户明确指定具体文件、子目录或任务需要核验来源时，才按最小范围读取相关内容。
- 如确需扫描上述目录，必须先说明目的、范围和预计成本，并优先使用精确路径或文件模式。

---

## 2. 文件与产物归属

### 2.1 正式文件

- 源码、配置、文档和测试只能放入项目已有的对应目录，并遵循现有命名风格。
- 测试应放在 `tests/`、`__tests__/` 或项目既有的 `*.test.*`、`*.spec.*` 位置。
- 公共逻辑应实现为可导入、可测试的模块；脚本只负责参数解析和流程编排。
- 禁止在根目录创建 `test.py`、`debug.py`、`quick_test.py`、`run_once.sh` 等临时文件。

### 2.2 临时文件

临时脚本、日志和输出只能放在：

```text
.tmp/agent/
  scripts/
  outputs/
  logs/
```

- 目录不存在时可以创建，但必须加入 `.gitignore`。
- 文件名统一为 `YYYYMMDD-HHMM-<task-slug>-<purpose>.<ext>`；`task-slug` 使用 2–5 个英文单词，`purpose` 使用 `repro`、`debug`、`benchmark`、`fixture-check` 等明确用途。
- 禁止 `tmp.txt`、`result.json`、`test2.py`、`new.py` 等含糊名称。
- 创建临时脚本前，优先使用已有测试、命令或 REPL。
- 临时脚本顶部必须说明用途、运行方式、创建原因和是否可删除；不得依赖机器专属绝对路径、写入源码目录或修改真实业务数据。
- 临时文件默认不提交、不作为正式实现依赖，并在任务完成后删除；确需保留时须说明路径、原因、清理方式和重新生成命令。

### 2.3 生成产物

- 产物应放入项目既有的 `build/`、`dist/`、`coverage/`、`reports/` 等目录，或 `.tmp/agent/outputs/`。
- 文件名必须表达任务和用途；除非属于项目既有规范，禁止在根目录生成 `output.txt`、`result.json`、`report.html` 等文件。
- 重复运行不得静默覆盖重要结果；论文相关产物应保留配置、输入和代码版本等来源信息。

---

## 3. Git 与改动边界

修改前和结束前均执行：

```bash
git status --short
```

- 区分用户已有改动、Agent 本次改动和工具自动生成文件。
- 工作区不是 Git 仓库时，明确说明无法进行版本状态检查。
- 不得执行破坏性回滚、静默删除失败测试、修改无关文件或提交真实密钥。
- 不得使用本机专属路径，或将 `.env`、token、cookie、账号信息写入代码、日志和产物。
- 最终回复须列出本次新增、修改、删除的全部文件。

---

## 4. 依赖、数据库与外部服务

### 4.1 新增依赖

新增依赖前必须说明：

1. 新依赖的必要性及现有替代方案。
2. 运行时、开发或测试范围及资源影响。
3. 修改的依赖清单和 lockfile。

不得仅为一次性脚本新增项目依赖。

### 4.2 数据库与外部系统

- 涉及数据库、迁移、外部 API、队列、缓存或云服务时，默认仅做 dry-run 或本地验证。
- 未经明确授权，不得修改或删除生产数据、发送外部消息或执行不可逆操作。
- 验证数据必须使用明确的测试命名，并说明清理方式。
- 原始数据和已发布的上游产物默认只读；新版本写入独立、命名明确的目录并记录来源关系。

---

## 5. 测试与验证

- 优先使用项目已有命令，如 `pytest`、`npm test`、`npm run lint`、`npm run typecheck`、`go test ./...` 或 `cargo test`。
- 修复缺陷时先增加可复现问题的回归测试；新增测试名称应描述行为而非实现细节。
- 测试应检查真实结果和关键不变量，不得只检查文件存在、数组非空或命令退出成功。
- 不得仅运行临时脚本就声称测试通过，也不得隐瞒失败项。
- 全量验证成本过高时，运行有代表性的子集，并说明未运行项、预计成本和剩余风险。
- 最终回复必须列出运行命令、通过与失败结果、失败是否与本次修改有关，以及未运行测试的原因。

## 6. AAAI 论文级系统工程约束

本节适用于论文方法、数据管线、模型、训练、推理、评测和图表。局部修复可按影响范围裁剪验证，但不得破坏以下约束。

### 6.1 研究目标与证据链

- 实现前明确研究问题或 SOP 步骤、论文假设、输入、输出和可验收结果；无法说明研究用途的功能不进入正式系统。
- 每项核心结论须能追溯到实现模块、配置、数据版本与划分、运行命令、指标和结构化产物。
- 明确区分原始观测、派生特征、模型输入、监督标签和仅分析字段；近似、合成数据、代理标签及缺失信息必须如实记录。
- 对既定论文方法或 SOP 的偏离必须说明原因、影响范围和验证方式，不得静默改变问题定义。

### 6.2 架构与数据契约

- 数据适配、预处理、几何变换、特征、标签、模型、训练、推理、评测和可视化按职责拆分，通过明确接口连接。
- 关键数据结构必须有稳定 schema，说明字段语义、来源、坐标系、单位、shape、dtype、有效范围和缺失值策略。
- 导入模块不得触发写入、训练或外部调用；避免巨型函数、隐式全局状态和循环依赖。
- 通过配置或适配器支持新记录、场景、划分和数据集；禁止硬编码样本编号、单一场景或只对第一条数据成立的逻辑。
- 坐标变换、插值、过滤、栅格化和掩码组合须定义并验证可检查的不变量。

### 6.3 配置化与可复现性

- 数据路径、时间窗口、采样率、坐标范围、分辨率、阈值、超参数、随机种子、数据划分和输出目录必须配置化。
- 所有随机过程使用可控种子；需跨进程或跨机器稳定时，不得依赖 Python 内置 `hash()`。
- 数据划分必须使用明确组键并检查 train/validation/test 重叠；同一轨迹、场景或主体不得泄漏到不同集合。
- 预处理统计量和可学习变换只能在训练集拟合，再应用于验证集和测试集。
- 关键产物须记录配置快照、生成时间、输入来源、数据与代码版本、样本计数和必要校验信息。
- 长耗时流程应提供断点恢复、原子写入或完整性检查，不得把部分产物视为成功结果。

### 6.4 数据、标签与泄漏防护

- 模型输入只能使用部署时真实可获得的信息；未来轨迹、未来占据、碰撞结果及其他真值只能用于标签或离线分析。
- 每个派生字段必须能追溯到原始字段和变换步骤，输入与标签侧应明确隔离。
- 数据构建后至少检查 schema、shape、dtype、NaN/Inf、数值范围、时间对齐、坐标系、掩码关系、划分重叠和样本计数。
- 对缺失传感器、无效姿态、过大插值跨度、空场景和越界目标制定明确策略；不得宽泛捕获异常后静默跳过。
- 评测须记录数据划分、样本数、过滤规则和失败样本，禁止只报告成功子集或挑选有利案例。

### 6.5 实验、指标与规模

- 研究功能应按需要提供配置、数据检查器、训练、推理、评测入口和复现命令，不能只交付模型主体或 Notebook。
- 为论文主张提供可执行的基线、消融和失败案例分析接口，不得靠复制代码或手工改变量完成消融。
- 指标须定义统计对象、聚合方式、单位、方向和边界情况，并用可人工核算的小样例验证。
- 随机性影响结论时，支持多种子运行，保存独立结果并报告均值、方差或置信区间。
- 定性图、表格和论文数字必须从结构化结果生成，避免手工编辑导致图文不一致。
- 验证顺序包括小型 fixture 单元测试、集成测试、真实记录 smoke test 和目标规模验证。
- 对大规模流程评估时间与空间复杂度，优先分块、流式、缓存或向量化，并记录环境、运行时、峰值资源或吞吐。
- 性能优化不得改变科学语义；优化前后须验证数值一致性或约定误差范围。

### 6.6 论文级完成标准

只有满足所有适用项，研究功能才可表述为“完成”：

1. 有明确、可复用的模块接口和命令行入口。
2. 有集中配置、确定性数据划分和随机种子控制。
3. 有数据 schema、坐标系、单位、字段来源和缺失值说明。
4. 有单元、集成、真实数据 smoke test、数据完整性检查及必要的规模验证。
5. 指标、图表、manifest 和评测报告可从结构化产物重新生成。
6. 有基线、消融、失败案例记录及论文主张到产物的证据链。
7. 已在目标规模运行，或明确列出未完成验证、成本和风险。

只提供 Notebook、一次性脚本、单样例或纯合成数据结果、硬编码结果、无配置模型、无评测数据流程，或无法从原始输入重建的图表，均不能单独作为论文系统交付。

## 7. 完成前自检

- [ ] 没有根目录临时文件、含糊命名文件或未说明的产物。
- [ ] 没有污染源码、残留调试输出、无用依赖、fixture 或空目录。
- [ ] 没有覆盖用户改动、无关格式化或静默改变配置与测试策略。
- [ ] 已运行与风险相称的测试，或说明未运行原因和剩余风险。
- [ ] 研究任务已检查数据契约、划分泄漏、未来信息、复现配置和产物证据链。
- [ ] 已清理临时文件，并列出全部改动文件。

---

## 8. 最终回复格式

```text
Summary:
- ...

Changed files:
- path/to/file: reason

Validation:
- command: result

Temporary files:
- None
```

如保留临时文件，还须逐项说明 `Reason`、`Safe to delete` 和重新生成方式；未运行测试时写明 `Not run: reason`。
